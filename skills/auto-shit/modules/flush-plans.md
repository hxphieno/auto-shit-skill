# flush-plans

spec/plan 文档生命周期管理。`writing-plans` 只管生成，`executing-plans` 只管执行，清理完全空白。此模块填补空白。

---

## 输入

- `docs/superpowers/specs/*.md`
- `docs/superpowers/plans/*.md`

---

## 前置检查

执行任何步骤之前，先确认文档目录存在。

- 检查 `docs/superpowers/` 目录是否存在。
- 若不存在 → 输出 `未发现 superpowers 文档目录` 并结束，不执行任何步骤。
- 若存在 → 按顺序执行以下步骤。

---

## 步骤 1：扫描文档

1. 使用 Glob 扫描 `docs/superpowers/specs/*.md` 和 `docs/superpowers/plans/*.md` 下所有文件。
2. 统计文件总数 N，记录每个文件的路径。
3. 对每个文件，使用 `stat -f %m`（macOS）或 `stat -c %Y`（Linux）获取 mtime，记录文件日期。
4. 列出所有文件及其日期，格式：`filename.md — mtime: YYYY-MM-DD`。
5. 若 N = 0 → 输出 `未发现 spec/plan 文档` 并结束。

---

## 步骤 2：提取标识符

对每个文档，Read 其完整内容并提取**具体标识符**。标识符是指代码中可以被 Grep 精确匹配的名称，不是模糊的关键词。

### 提取目标

| 类别 | 示例 |
|------|------|
| 函数名 / 方法名 | `createUser`, `handleSubmit`, `flush_memory` |
| API 路径 | `/api/v1/users`, `POST /auth/login` |
| 变量名 / 常量名 | `MAX_RETRY_COUNT`, `defaultConfig` |
| 组件名 | `<UserProfile>`, `FlushButton` |
| 类名 / 接口名 | `SkillRouter`, `IMemoryStore` |
| 文件名（被 spec 明确指定的） | `flush-plans.md`, `settings.local.json` |
| 命令名 / CLI 子命令 | `auto-shit scan`, `flush --all` |

### 提取规则

1. **只提取被 spec 明确列出或定义的标识符。** 不要从叙述性段落中猜测可能的标识符。
2. **代码块中的标识符优先级最高** — 在 `` ``` `` 代码块中出现的函数名、类名等是最可靠的标识符。
3. **表格和列表中的标识符次之** — 在表格、项目列表中以 `code font` 出现的名称。
4. **叙述性文字中的标识符最低** — 仅当文字中使用了 `code font` 标记时才提取。
5. 对每个文档，记录提取到的标识符列表及其来源行号。
6. 若某文档为纯叙述性文本（无代码块、无 `code font` 标记），记录为"无可靠标识符"，后续步骤中标注准确度有限。

---

## 步骤 3：代码级实现验证

对每个提取到的标识符，使用 Grep 在当前代码库中搜索其存在。

### 搜索策略

1. **搜索范围：** 整个项目目录，排除以下路径：
   - `docs/` — 文档中引用自身不算实现
   - `node_modules/`, `.git/`, `dist/`, `build/` — 依赖和构建产物
2. **匹配方式：** 精确字符串匹配（非正则），区分大小写。
3. **判定标准：**
   - 标识符在代码文件（`.ts`, `.js`, `.py`, `.go`, `.rs`, `.java`, `.md` 除外的源码文件）中存在 → 视为"已实现"
   - 标识符仅在文档中存在 → 不算实现
4. **按文档汇总：** 对每个 spec/plan 文档，统计其标识符的匹配情况：
   - `匹配数 / 总标识符数`（例如 `8/10 标识符已在代码中找到`）

### 为什么不用 git log

Git commit message 通常是 "fix typo"、"wip"、"update" 这类模糊信息，不会包含 spec 中定义的具体函数名或 API 路径。代码文件中是否存在该标识符是更可靠的实现证据。

---

## 步骤 4：分类

根据步骤 2-3 的结果，将每个文档分为三类：

### 已实现

满足以下条件：
- 代码库中找到了该文档 **超过半数** 标识符的匹配

处理建议：归档到 `docs/superpowers/archived/`

### 明显过期

满足以下 **两个条件同时成立**：
- 文档 mtime 距今 **超过 180 天**
- 代码库中 **未找到任何** 匹配标识符（或文档无可靠标识符且超过 180 天）

处理建议：建议删除

### 活跃

不满足以上两类条件的所有文档：
- 标识符部分匹配（实现进行中）
- 标识符未匹配但文档较新（< 180 天）
- 无可靠标识符但文档较新

处理建议：保留，不做任何操作

---

## 步骤 5：用户确认

输出分类清单，等待用户逐条确认。

### 输出格式

```
[flush-plans 分类结果]

📦 已实现（建议归档）：
  1. specs/user-auth-spec.md — 8/10 标识符匹配，mtime: 2025-09-15
  2. plans/api-v2-plan.md — 5/6 标识符匹配，mtime: 2025-10-01

🗑️ 明显过期（建议删除）：
  3. specs/old-feature-spec.md — 0/4 标识符匹配，mtime: 2025-01-20（超过 180 天）
  4. plans/abandoned-plan.md — 无可靠标识符，mtime: 2024-12-05（超过 180 天）

✅ 活跃（保留）：
  5. specs/current-feature.md — 3/8 标识符匹配，mtime: 2026-03-10
  6. plans/wip-plan.md — 0/5 标识符匹配，mtime: 2026-04-01

---
请逐条确认"已实现"和"明显过期"文档的处理方式。
```

### 确认规则

1. **"已实现"文档必须逐条确认** — 不允许批量自动归档。逐个询问用户：`归档 {filename} 到 archived/？(y/n)`
2. **"明显过期"文档也需逐条确认** — 逐个询问用户：`删除 {filename}？(y/n)`
3. **"活跃"文档不需确认** — 直接保留。
4. 用户拒绝某条 → 跳过，不再重复询问。

---

## 步骤 6：执行

对用户确认的每个文档执行对应操作：

### 归档"已实现"文档

1. 检查 `docs/superpowers/archived/` 目录是否存在，不存在则创建。
2. 将文档移动到 `docs/superpowers/archived/` 目录（使用 `mv`）。
3. 若目标路径已存在同名文件 → 在文件名后追加日期后缀（如 `spec.md` → `spec-2026-04-07.md`）。

### 删除"明显过期"文档

1. 直接删除文件（使用 `rm`）。

---

## 输出格式

所有步骤执行完毕后，输出汇总报告：

```
[flush-plans 完成]
- 扫描了 {N} 份文档
- 已实现（已归档）：{M} 份
- 明显过期（已删除）：{K} 份
- 活跃（保留）：{L} 份
```

若所有文档均为活跃状态 → 输出 `所有文档均为活跃状态`。

若存在无可靠标识符的文档 → 在报告末尾追加：
```
注意：{X} 份文档为纯叙述性内容，标识符提取准确度有限，分类结果供参考。
```

---

## 边界情况

- `docs/superpowers/` 目录不存在 → 输出 `未发现 superpowers 文档目录` 并结束。
- specs/ 或 plans/ 子目录不存在 → 仅扫描存在的子目录。若两者都不存在但 superpowers/ 存在 → 输出 `未发现 spec/plan 文档` 并结束。
- 当前目录不是 git 仓库 → 跳过任何 git 相关操作，仅使用文件日期作为辅助判断依据。
- 所有文档均为活跃状态 → 输出 `所有文档均为活跃状态`。
- 纯叙述性 spec（无代码块、无 `code font` 标记） → 记录为"无可靠标识符"，在输出中标注准确度有限。
- archived/ 目录中已有同名文件 → 追加日期后缀避免覆盖。
